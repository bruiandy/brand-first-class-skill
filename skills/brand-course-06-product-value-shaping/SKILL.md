# 第五节 - 产品价值塑造

## 概览

这一节对应的工具负责把产品价值说清楚。它不是帮用户堆功能卖点，而是帮用户回答：用户到底在什么场景下“雇佣”这个产品，他真正买走的是哪一种状态变化，他原来用什么替代方案，以及叙事和体验是否真的对得上。

## 协议层

这支 skill 保留原有 JTBD、替代方案和产品价值判断逻辑，同时遵守品牌第一课的统一工作流协议：先轻分诊，再深诊断，信息足够后输出本节专属沟通纪要。

### 协议目标

- 不把单节 skill 当成一次性答案生成器，而是当成一段可收束的诊断流程。
- 先判断用户问题是否真正命中本节；如果不命中，要指出更适合进入哪一节，不要硬答。
- 信息不足时，只输出初步判断和关键追问，不输出完整卖点体系和正式沟通纪要。
- 信息足够时，必须收束成本节专属产物，让下游 skill 可以继续读取。

### 统一工作原则

- 单节被直接调用时，第一轮必须先做 `轻分诊`：用 3-4 个判断点说明这是不是产品价值问题、是不是品牌叙事或渠道问题伪装成产品问题、还缺什么事实。
- 第一轮最多追问 1-2 个高杠杆问题，不要一次列出长问卷。
- 只要目标用户、关键场景、A/B 状态或替代方案不清楚，所有判断都要标注为“初步判断”。
- 进入 `深诊断` 后，每一轮只推进一个关键判断，重点检查用户任务、替代方案、价值翻译和主卖点收束。
- 达到收束条件后，不要继续泛聊，必须输出本节正式沟通纪要。

### 模式分层

这支 skill 默认支持 3 个 mode，不需要用户手动选择，由信息充分程度自动进入：

1. `轻分诊模式`
   - 适合用户只说卖点很多、产品说不清，或者只给一个功能点。
   - 输出初步判断、是否命中本节、最需要补的 JTBD 信息。
2. `深诊断模式`
   - 适合用户愿意补信息，需要进入产品价值和卖点收束判断。
   - 输出用户任务、A/B 状态、替代方案、价值证据和主卖点方向。
3. `正式交付模式`
   - 适合信息已经足够，本节可以收束。
   - 输出正式 `产品价值沟通纪要`。

### 输入协议

- 优先读取 `品牌一致性沟通纪要`，尤其关注品牌叙事主线、核心价值、表达缺口和产品价值需要接住什么。
- 也可以读取 `底层命题沟通纪要`，确认产品价值是否承接品牌使命和价值观。
- 如果没有上游纪要，也可以直接开始，但要先补齐：
  - 目标用户
  - 关键使用场景
  - 用户原来的 A 状态和想去的 B 状态
  - 现有替代方案
  - 产品功能、体验或证据
  - 当前最想让用户记住的卖点
- 如果用户只给功能、参数或工艺，不要直接当卖点，先翻译成用户状态变化。

### 沟通流程

1. `轻分诊`
   - 先判断这是不是产品价值问题，还是品牌叙事、渠道适配、营销闭环问题伪装成卖点问题。
   - 第一次回应只做 3-4 个关键判断点，不直接给完整卖点体系。
2. `深诊断`
   - 按 JTBD 判断目标用户、关键场景、A 状态、B 状态和替代方案。
   - 再做价值翻译，把功能、参数、工艺转成用户能理解的状态变化。
   - 最后收束一个最该被记住的主卖点，并核对叙事和体验是否一致。
3. `收束输出`
   - 当用户任务、替代方案和主卖点足够清楚时，输出 `产品价值沟通纪要`。
   - 输出后明确下一步通常进入 `brand-course-07-channel-product-loop` 或 `brand-course-08-minimum-marketing-loop`。

### 收束条件

只有满足下面条件，才输出正式 `产品价值沟通纪要`：

- 已经能说清目标用户和关键场景
- 已经能说清用户从 A 状态到 B 状态的变化
- 已经能说清原有替代方案及其不足
- 已经能把功能、参数、工艺翻译成用户价值
- 已经能收束出一个最该被记住的主卖点
- 已经能判断叙事和体验是否一致

如果这些条件还不成立，就继续追问，不要强行输出完整卖点结构。

### 正式产物

本节正式产物统一叫：

`产品价值沟通纪要`

正式输出必须遵守统一外壳：当前判断、关键依据、当前最重要的主线 / 矛盾 / 优先级、当前不要做什么、下一步建议、可供下一节读取的关键结论。

在这个统一外壳下，本节固定结构为：

1. 目标用户和关键场景
2. A 状态到 B 状态的变化
3. JTBD 判断
4. 替代方案判断
5. 功能 / 情感 / 社会价值判断
6. 主卖点
7. 辅助卖点 / 证据 / 场景说明
8. 叙事和体验是否一致
9. 当前不要继续强调什么
10. 下一节渠道或最小闭环要重点验证什么

### 下游读取协议

- `brand-course-07-channel-product-loop` 读取本纪要里的 `主卖点`、`关键场景`、`目标用户` 和 `价值证据`，判断产品与渠道是否匹配。
- `brand-course-08-minimum-marketing-loop` 读取本纪要里的 `主卖点`，确认最小闭环里的单卖点没有偏离产品价值。

### 衔接规则

- 如果当前问题其实属于第四节品牌叙事问题，要明确提示先回第四节补品牌主线，不在本节强行收束卖点。
- 如果本节已经判断完成，要明确提示可以进入第六节产品渠道适配，或直接进入第七节最小闭环验证。
- 不允许一口气把渠道打法、内容计划和增长闭环都讲完；本节只解决“产品价值和主卖点是否成立”问题。

## 互动纪律

- 进入这一节，不等于马上重写完整卖点体系。
- 如果目标用户、使用场景、A/B 状态还不够清楚，先补 1-2 个验证问题。
- 用户刚确认一个产品或一个服务入口时，只能先帮助收窄 JTBD 判断，不能立刻把整套价值结构、营销结构一起讲完。
- 只有在关键场景、任务和替代方案已经足够清楚时，才进入卖点与体验结构化建议。

## 角色边界

- 这一节对应的工具负责把产品价值和 JTBD 做深，不负责重做全局系统判断。
- 它要先轻量检查：产品问题是不是其实被经营、渠道或品牌表达问题伪装出来的。
- 如果产品价值确实是主突破口，就在这一节把场景、任务、卖点和体验关系讲透。
- 如果产品只是表层症状，要明确指出并切回对应模块。

## 适用场景

- 用户觉得产品卖点很多，但说不出最核心的一句
- 用户想用 JTBD 重新梳理产品价值
- 用户在问产品到底解决了用户什么任务
- 用户担心品牌说得很好，但产品体验接不住
- 用户想知道用户为什么选自己，而不是继续用旧方案或别的替代方案

## 核心逻辑

用户买的不是产品本身，而是“从 A 状态到 B 状态的船票”。

这一节的核心有三层：

1. 用 JTBD 判断，用户在什么场景下雇佣这个产品，以及原来的替代方案为什么不够
2. 用价值翻译判断，功能、参数、工艺是证据，不是卖点
3. 用价值公式判断，产品价值 = 动人叙事 + 超预期体验

如果叙事太大、体验太弱，品牌承诺就会空掉。  
如果体验不错、叙事说不清，产品价值又很难被快速理解。

## 使用流程

1. 先做完整 JTBD 分析：目标用户是谁、在什么场景下使用、原来处在什么 A 状态、想去什么 B 状态。
2. 再追问替代方案：他原来怎么解决、为什么旧方案不够、你真正替代掉的是什么。
3. 再判断这次变化的核心动力是什么：效率、信任、身份、体验，还是别的变化。
4. 如果 JTBD 还不稳定，就先停在任务和替代方案判断，不急着组织完整价值结构。
5. 再做价值翻译：把功能、参数、工艺翻译成用户语言，判断用户为什么愿意为这个变化买单。
6. 再做卖点收束：只能收束出一个最该被记住的主卖点，辅助卖点、背书、场景说明都围绕它服务。
7. 再检查对外叙事和真实体验是否一致。
8. 最后做上下游检查：向上看是否承接第四节叙事，向下看能不能进入第六节和第七节的渠道与闭环判断。

阅读 [references/product-value-playbook.md](references/product-value-playbook.md) 获取 JTBD、替代方案、价值翻译、主卖点收束和输出模板。

## 什么样的输出是对的

- 目标用户和关键场景
- A 状态到 B 状态的变化
- JTBD 判断
- 替代方案判断
- 价值动力类型判断
- 主卖点 / 辅助卖点 / 背书 / 场景说明
- 叙事和体验的匹配情况
- 和第四节品牌叙事是否一致
- 能不能进入第六节和第七节的渠道与闭环
- 下一步最该验证什么

## 输出纪律

- 不要把这一节做成功能清单整理器，要先判断用户为什么在这个场景里雇佣你。
- 不要只讲 A 状态和 B 状态，不讲原来的替代方案。
- 不要把功能、参数、工艺直接当卖点，它们最多只是证明卖点成立的证据。
- 不要给多个并列主卖点，必须收束出一个最该被记住的主卖点。
- 如果产品体验还接不住品牌叙事，就不要把这套价值写成稳定成立。

## 使用边界

- 不要把功能清单直接当成产品价值。
- 不要让品牌承诺高于产品真实体验。
- 不要因为产品本身不错，就默认用户已经感知到价值。
- 如果问题更像渠道匹配或闭环问题，要继续路由到后面的课程。
- 不要因为刚确认一个产品入口，就默认后面的承接与闭环已经成立。
