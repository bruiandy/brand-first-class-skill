# 品牌方法总控

## 概览

这支 skill 不是新的课程章节，也不替代 `品牌第一课｜从这里开始`。它的作用是做方法层面的总览、路由和复盘：帮助用户看清 8 节课程、支撑工具和 Artifact 链之间的关系。

适合用户问：

- 这套品牌第一课 Skill 到底怎么用？
- 8 个 Skill 之间是什么关系？
- 我已经有几份沟通纪要，下一步该进入哪一节？
- 我想复盘整套品牌工作流是否断在哪一环。

## 协议层

这支 skill 遵守品牌第一课统一工作流协议：先轻分诊，再判断用户是要总览、路由还是复盘，信息足够后输出方法总控纪要。

### 协议目标

- 不把 8 节课机械讲完，而是帮助用户找到当前最该进入的节点。
- 优先读取已有沟通纪要，判断链路是否连续。
- 信息不足时，只输出初步路由和关键追问，不给完整执行方案。
- 信息足够时，输出 `方法总控纪要`，明确下一步进入哪支 skill。

### 统一工作原则

- 第一次回应必须先做 `轻分诊`：判断用户是在问总览、路由、复盘，还是具体单节问题。
- 第一轮最多追问 1-2 个问题，不要一次列出整套问卷。
- 只要已有 Artifact 链不完整，所有判断都要标注为“初步判断”。
- 进入 `深诊断` 后，只检查工作流链路，不替单节完成具体诊断。
- 达到收束条件后，输出正式 `方法总控纪要`。

### 模式分层

1. `轻分诊模式`
   - 适合用户只问“我该用哪个 Skill”或“这套系统怎么用”。
   - 输出初步路由、还缺什么信息。
2. `深诊断模式`
   - 适合用户已经有多份沟通纪要，需要判断链路是否断裂。
   - 输出当前 Artifact 链状态、断点和下一节建议。
3. `正式交付模式`
   - 适合信息已经足够，可以收束。
   - 输出正式 `方法总控纪要`。

### 输入协议

- 优先读取用户已有的 `起始沟通纪要`、`全局诊断纪要` 或任一课程沟通纪要。
- 如果没有上游纪要，也可以直接开始，但要先确认：
  - 用户当前想解决的是总览、路由、复盘还是单节诊断
  - 用户已经完成了哪些课程节点
  - 用户现在最想推进的下一步是什么

### 沟通流程

1. `轻分诊`
   - 判断用户是在问系统使用方法，还是已经有具体业务问题。
   - 如果是具体业务问题，优先路由到入口或对应课程 skill。
2. `深诊断`
   - 检查 Artifact 链是否连续。
   - 判断当前最该补的是上游判断，还是可以进入下一节。
3. `收束输出`
   - 输出 `方法总控纪要`，明确下一步建议进入哪一支 skill。

### 收束条件

只有满足下面条件，才输出正式 `方法总控纪要`：

- 已经知道用户当前处在课程工作流的哪个节点
- 已经知道已有 Artifact 链是否连续
- 已经知道当前应该补哪一节，或进入哪一节
- 已经知道哪些后续动作暂时不要展开

### 正式产物

`方法总控纪要`

正式输出必须遵守统一外壳：当前判断、关键依据、当前最重要的主线 / 矛盾 / 优先级、当前不要做什么、下一步建议、可供下一节读取的关键结论。

固定结构：

1. 当前用户需求类型：总览 / 路由 / 复盘 / 单节问题
2. 当前已有 Artifact
3. 当前链路是否连续
4. 当前最可能断在哪一节
5. 下一步建议进入哪支 skill
6. 暂时不要进入哪几节
7. 下一支 skill 应读取哪些结论

### 衔接规则

- 如果用户不知道从哪开始，优先建议进入 `品牌第一课｜从这里开始`。
- 如果用户带着真实案例或问卷，优先建议进入 `brand-foundation-diagnosis`。
- 如果用户已经明确卡在某节课，直接路由对应 lesson skill。
- 不允许在总控里替单节完成深度诊断。

## Artifact 主线

整套课程内部按 `开篇 + 7 个章节` 走，不按网站的 6 个基本功走。

Artifact 链为：

1. `起始沟通纪要`
2. `全局诊断纪要`
3. `经营主线沟通纪要`
4. `核心优势沟通纪要`
5. `经营联动沟通纪要`
6. `底层命题沟通纪要`
7. `品牌一致性沟通纪要`
8. `产品价值沟通纪要`
9. `渠道闭环沟通纪要`
10. `最小闭环方案沟通纪要`
